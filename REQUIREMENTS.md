# 멀티 사용자 AI 채팅 웹 서비스 요구사항

## 1. 문서 목적

관리자가 생성한 사용자들이 관리자가 허용한 AI API를 통해 대화할 수 있는 웹 서비스를 구축한다. 각 사용자의 계정, 대화방, 메시지 및 컨텍스트는 서로 완전히 분리하며, 서비스는 Linux 서버에서 운영한다.

이 문서는 초기 개발 범위를 정의한 초안이다. 아직 정해지지 않은 값은 **검토 필요**로 표시한다.

명세의 확정 범위와 남은 결정사항은 [SPEC_STATUS.md](./SPEC_STATUS.md)에서 관리한다.

구현 기술의 권장 조합과 영역별 대체안은 [TECH_STACK_OPTIONS.md](./TECH_STACK_OPTIONS.md)에서 관리한다.

전체 명세의 누락·불일치 점검 결과는 [SPEC_AUDIT.md](./SPEC_AUDIT.md)에서 관리한다.

확정된 1차 서버 조건과 공개 인터넷 운영 기준은 [DEPLOYMENT_PROFILE.md](./DEPLOYMENT_PROFILE.md)에서 관리한다.

port, 관리자 계정과 서버 시작 설정 파일은 [SERVER_CONFIG_SPEC.md](./SERVER_CONFIG_SPEC.md)를 따른다.

다음 개발 단계부터 코드와 함께 상세 문서를 계속 작성·갱신하는 절차는 [DEVELOPMENT_WORKFLOW.md](./DEVELOPMENT_WORKFLOW.md)를 따른다. 기능 구현 시 관련 상세 문서 갱신을 완료 조건에 포함한다.

## 2. 핵심 요구사항 요약

- 고정된 최초 관리자 계정 1개를 둔다.
- 공개 회원가입 기능은 제공하지 않는다.
- 관리자가 활성화한 경우 공유 코드를 통해 서로 격리된 임시 게스트 체험 session을 발급한다.
- 관리자가 사용자 계정을 생성·수정·비활성화하고 API 사용 권한을 부여한다.
- 관리자가 서비스 제공자를 선택하고 자격증명을 등록할 수 있다.
- API 키 방식 제공자는 제공자 선택과 API 키 입력만으로 모델을 자동 조회하고 기본 설정으로 사용할 수 있어야 한다.
- LLM Gateway를 포함해 GPT·Gemini·Claude 모델을 제공하는 서비스들을 지원한다.
- 각 사용자는 관리자가 허용한 제공자와 모델만 사용할 수 있다.
- 사용자별 대화방, 메시지, 첨부파일 및 대화 컨텍스트를 분리한다.
- 텍스트 입력과 TXT, PDF, 이미지 첨부를 지원한다.
- 이미지 생성 기능은 제공하지 않는다.
- 사용자가 허용된 범위 안에서 모델 요청 파라미터를 조정할 수 있다.
- Linux 서버에 배포하여 웹 브라우저로 사용한다.

## 3. 사용자 역할 및 권한

### 3.1 관리자

최초 관리자 계정은 서버 구동 파일과 같은 배포 폴더의 `config.yaml`에서 직접 지정한다. 항목은 `admin.username`, `admin.passwordHash`, `admin.totpSecret`, `admin.requireTotp`이며 읽기 권한을 `0600`으로 제한하고 container에는 read-only로 mount한다. 환경변수는 `APICHAT_CONFIG_FILE`로 설정 파일 경로를 바꾸는 용도로만 사용한다. 평문 비밀번호와 TOTP secret을 Git 저장소나 프론트엔드 설정에 기록해서는 안 된다. 서버 시작 시 설정값을 기준으로 고정 관리자 계정을 생성하거나 동기화한다.

배포 폴더에는 관리자 비밀번호를 숨김 입력으로 받아 Argon2id hash로 변환하고 `config.yaml`을 갱신하는 `apichat-admin` 계정 설정 도구를 포함한다. 이 도구는 상시 network daemon이 아니라 필요할 때만 실행하는 CLI 또는 일회성 container로 제공한다.

관리자는 다음 작업을 수행할 수 있다.

- 사용자 계정 생성
- 사용자 정보 및 비밀번호 변경
- 사용자 계정 활성화·비활성화
- 사용자 계정 관리 화면에서 새 비밀번호 설정
- AI API 공급자, 엔드포인트, 인증키 및 모델 등록·수정·비활성화
- 사용자별 사용 가능 API와 모델 지정
- 사용자별 사용 가능 파라미터와 허용 범위 지정
- 사용자별 사용량 및 오류 기록 조회
- 필요시 사용자의 대화 데이터 삭제

관리자 계정 자체를 일반 관리 화면에서 삭제하거나 이름·비밀번호를 변경할 수 없도록 한다. 관리자 계정 변경은 서버 설정에서만 수행한다.

사용자 대화 본문 열람 기능은 낮은 우선순위의 선택 기능으로 둔다. 구현할 경우 열람 권한을 고정 관리자에게만 허용하고, 열람 사유 입력과 감사 로그를 필수로 한다.

### 3.2 일반 사용자

일반 사용자는 다음 작업을 수행할 수 있다.

- 관리자가 발급한 계정으로 로그인·로그아웃
- 자신에게 허용된 API와 모델 선택
- 새 대화방 생성, 이름 변경 및 삭제
- 자신의 대화방과 메시지 조회
- TXT, PDF 및 이미지 파일 첨부
- 허용된 모델 파라미터 설정
- 답변 생성, 생성 중단 및 재생성

다른 사용자의 계정, 대화, 첨부파일, API 키 및 사용량 정보에는 접근할 수 없다.

### 3.3 게스트

게스트는 사용자 계정이나 회원가입 없이 관리자가 공유한 게스트 코드를 입력해 독립된 임시 session에 참가한다. 같은 코드를 사용해도 접속자마다 별도 임시 주체를 발급하며 서로의 대화·메시지·첨부파일과 컨텍스트를 공유하지 않는다. 상세 수명, 권한, 호출 제한과 삭제 정책은 [GUEST_ACCESS_SPEC.md](./GUEST_ACCESS_SPEC.md)를 따른다.

## 4. 인증 및 사용자 관리

- 공개 회원가입 페이지와 회원가입 API를 제공하지 않는다.
- 인터넷 공개 배포에서는 고정 관리자 로그인의 TOTP MFA를 필수로 한다.
- 관리자 MFA 복구 code는 일회용으로 발급해 offline 보관하며 사용 후 폐기한다.
- 비밀번호는 평문으로 저장하지 않고 Argon2id 또는 bcrypt로 해시하여 저장한다.
- 관리자 비밀번호는 최소 10자와 필수 TOTP를 함께 적용하고 일반 사용자 비밀번호는 최소 8자로 한다.
- 로그인 실패 횟수 제한 또는 지연 처리를 적용한다.
- 세션 쿠키는 `HttpOnly`, `Secure`, `SameSite` 속성을 사용한다.
- 계정당 활성 세션은 최대 3개로 제한한다.
- 4번째 로그인 성공 시 마지막 사용 시각이 가장 오래된 세션을 자동 종료한다.
- 세션은 24시간 동안 활동이 없으면 만료하고, 계속 사용하더라도 로그인 후 최대 7일이 지나면 다시 로그인하게 한다.
- 사용자 비밀번호 변경, 계정 비활성화 또는 삭제 시 해당 사용자의 기존 세션을 모두 즉시 만료시킨다.
- 고정 관리자 설정이 변경되면 기존 관리자 세션을 모두 만료시킨다.
- 관리자 작업은 감사 로그로 남긴다.
- 게스트 체험은 기본 비활성이고 공유 코드가 설정된 경우에만 관리자가 활성화할 수 있다.
- 게스트 공유 코드는 최소 6자, 최대 128자로 하고 Argon2id hash만 저장한다.
- 게스트 session은 기본 1시간 idle·24시간 absolute 만료를 적용하고 로그아웃·만료 후 임시 데이터를 삭제한다.

## 5. AI API 및 모델 관리

AI 공급자 어댑터, 모델 기능 감지, 요청 형식 선택과 파라미터 변환은 [AI_INTEGRATION_SPEC.md](./AI_INTEGRATION_SPEC.md)를 따르고, 제공자 선택·자격증명·자동 설정과 고급 설정은 [PROVIDER_REGISTRATION_SPEC.md](./PROVIDER_REGISTRATION_SPEC.md)를 따른다. 관리자 로그 화면과 보관·마스킹 정책은 [ADMIN_LOGGING_SPEC.md](./ADMIN_LOGGING_SPEC.md)를 따른다.

### 5.1 API 등록

첫 버전은 `provider-manager-v1.10.0.js`의 provider registry 구조를 참고하여 복수 서비스 제공자를 지원한다. 기본 제공자 템플릿에는 LLM Gateway를 포함하며 관리자가 제공자를 선택하고 API 키만 입력해 빠르게 등록할 수 있게 한다.

관리자는 다음 정보를 등록할 수 있다.

- 표시 이름
- 서비스 제공자 또는 사용자 정의 provider template
- API 기본 URL
- API 키 또는 인증 정보
- 모델 ID와 사용자에게 표시할 모델명
- 텍스트, 이미지, 파일 등 모델별 지원 기능
- 활성화 상태
- 요청 제한시간
- 컨텍스트 길이 및 최대 출력 토큰

API 키는 데이터베이스에 평문으로 저장하지 않고 서버 전용 암호화 키로 암호화한다. API 키 및 복호화 키는 브라우저로 전달하거나 로그에 기록하지 않는다.

임의의 URL 등록은 서버 내부망 접근에 악용될 수 있으므로 SSRF 방어가 필요하다. 사설 IP, 메타데이터 주소 및 허용되지 않은 프로토콜 접근을 차단하거나 관리자 설정에서 허용 도메인 목록을 운영한다.

### 5.2 사용자별 권한

- 사용자와 API·모델 간 다대다 권한 관계를 저장한다.
- 권한이 회수되면 새 요청부터 즉시 사용할 수 없게 한다.
- 기존 대화가 권한을 잃은 모델을 사용한 경우 기록은 조회할 수 있지만 추가 요청은 차단한다.
- 모델별로 사용량 한도를 둘 수 있도록 확장 가능한 구조를 권장한다.
- 관리자는 사용자 계정 전체와 모델별 일일 호출 제한을 설정할 수 있다. `null`은 무제한이다.
- 호출 제한은 timezone 기준 날짜별로 DB에서 원자적으로 집계하며 권한 또는 제한 변경은 다음 요청부터 적용한다.

### 5.3 공급자 호환성

첫 버전에서는 OpenAI Chat Completions, OpenAI Responses, Anthropic Messages와 Gemini GenerateContent 규격을 지원한다. 제공자 템플릿이 기본 URL, 인증 방식, 모델 조회 경로와 지원 규격을 지정하고 서버가 모델별 요청 형식을 선택한다.

모든 API가 스트리밍, 이미지 입력, system 메시지 또는 동일한 파라미터를 지원하는 것은 아니다. 서버가 공급자별 차이를 변환하고, 지원하지 않는 기능은 화면에서 비활성화해야 한다.

`provider-manager-v1.10.0.js`에 존재하는 제공자 템플릿, 모델 조회, 스트리밍, thinking 표시, 일괄 출력, 요청 형식 선택, 추론 설정, 캐시, 키 그룹, 오류 재시도, 요청 로그와 모델별 파라미터 보정 구조를 참고한다. RisuAI 저장소·IPC·공개 프록시처럼 웹 서버에 맞지 않는 기반 기능은 서버용 구현으로 대체하거나 제외한다.

## 6. 대화 및 컨텍스트 관리

- 사용자는 여러 개의 대화방을 가질 수 있다.
- 각 대화방은 반드시 한 사용자에게 귀속된다.
- 메시지에는 역할, 내용, 사용 모델, 파라미터, 생성 시각, 사용 토큰 및 오류 상태를 저장한다.
- 답변은 가능하면 스트리밍 방식으로 화면에 표시한다.
- 대화 컨텍스트는 현재 대화방의 메시지만 사용하며 다른 대화방이나 다른 사용자의 데이터를 자동으로 섞지 않는다.
- 사용자는 대화방별 system prompt를 직접 작성하고 수정할 수 있다.
- 사용자는 대화방별 기본 모델과 생성 파라미터를 독립적으로 저장할 수 있다. 대화 제목이 같아도 대화 UUID가 다르면 설정을 공유하지 않는다.
- 사용자는 이전 대화 전송 범위를 완료된 user-assistant 대화 턴 수로 선택한다. 기본값 `0`은 현재 분기의 이전 대화 전체를 의미한다.
- 사용자는 대화방별 컨텍스트 토큰 목표값을 설정한다. 기본값은 100,000토큰이며 실제 적용값은 선택 모델의 최대 입력 한도를 넘을 수 없다.
- 예상 입력이 컨텍스트 토큰 목표값을 넘으면 오래된 대화부터 요약하여 컨텍스트에 포함한다. 원본 메시지는 삭제하지 않는다.
- 관리자는 요약 전용 모델과 요약 프롬프트를 지정한다. 요약 실패 시 본 요청을 임의로 잘라 보내지 않고 사용자에게 오류와 재시도 방법을 안내한다.
- 사용자는 첨부문서를 이후 메시지에도 계속 포함할지 첨부 단위로 선택할 수 있다.
- 대화 중 모델을 변경해도 같은 대화를 이어간다. 새 모델에 지원되지 않는 파라미터는 제거하고 사용자에게 알린다.
- 답변 재생성은 기존 답변을 덮어쓰지 않고 메시지 분기로 보존한다. 이후 컨텍스트에는 사용자가 선택한 활성 분기만 포함한다.
- 사용자 계정 삭제 시 소유한 대화, 메시지, 요약, 원본 첨부파일, 추출 데이터 및 세션을 삭제한다.

### 6.1 컨텍스트 격리 원칙

모든 대화·메시지·첨부파일 조회와 수정 요청은 서버에서 현재 로그인 사용자의 소유권을 검사한다. 클라이언트가 전달한 사용자 ID만 신뢰하지 않는다.

관리자 대화 본문 열람 기능은 2차 범위로 둔다. 기능이 비활성화된 동안 관리자는 운영 정보와 사용량만 조회할 수 있다.

## 7. 파일 및 이미지 처리

### 7.1 텍스트 계열 파일

- UTF-8, UTF-8 BOM, UTF-16 및 CP949 인코딩을 감지하고 텍스트로 변환한다.
- `.txt`, `.md`, `.markdown`, `.json`, `.jsonl`, `.csv`, `.tsv`, `.log`, `.xml`, `.yaml`, `.yml`을 지원한다.
- 일반적인 소스 코드인 `.js`, `.ts`, `.jsx`, `.tsx`, `.py`, `.java`, `.c`, `.cpp`, `.h`, `.hpp`, `.cs`, `.go`, `.rs`, `.php`, `.rb`, `.sh`, `.ps1`, `.sql`, `.html`, `.css`를 텍스트 파일로 지원한다.
- GIF, HEIC, DOCX 및 실행·바이너리 파일은 지원하지 않는다.
- 파일 내용을 추출하여 대화 컨텍스트에 포함한다.
- 확장자, 실제 MIME 형식, 바이너리 포함 여부와 문자 수를 서버에서 검증한다.

### 7.2 PDF 파일

- 텍스트가 포함된 일반 PDF에서 본문을 추출한다.
- PDF는 파일당 최대 100페이지로 제한한다.
- 암호화되었거나 손상된 PDF는 명확한 오류를 표시한다.
- 스캔 이미지로만 구성된 PDF는 서버 로컬 OCR로 한국어·영어 텍스트를 추출한다.

OCR(Optical Character Recognition, 광학 문자 인식)은 종이에 인쇄된 문서를 사진처럼 스캔한 PDF에서 글자 모양을 분석해 검색·복사 가능한 텍스트로 변환하는 기능이다. 일반 PDF는 글자 데이터가 들어 있어 바로 추출하며, 스캔 PDF는 서버 로컬 한국어·영어 OCR을 실행한다. 처리 페이지 수를 표시하고 복잡한 표·필기·저해상도 문서는 인식 정확도가 낮을 수 있음을 전제로 한다.

### 7.3 이미지 파일

- JPEG, PNG, WebP를 우선 지원한다.
- 이미지 이해 기능이 있는 멀티모달 모델을 선택했을 때만 첨부할 수 있다.
- 확장자뿐 아니라 실제 MIME 형식도 검사한다.
- 이미지 크기 및 해상도 제한을 적용한다.
- EXIF 등 불필요한 메타데이터 제거를 권장한다.

### 7.4 공통 보안

- 파일 하나의 최대 크기는 10MB이다.
- 메시지 하나에 첨부할 수 있는 파일은 최대 10개이다.
- 원본 파일과 추출된 텍스트를 모두 서버에 저장한다.
- 첨부파일의 기본 보관 기간은 업로드 후 30일이며 관리자가 전체 기본값을 변경할 수 있다.
- 보관 기간이 끝나면 원본 파일과 추출 데이터는 삭제하고 메시지에는 만료된 첨부파일의 이름과 메타데이터만 남긴다. 이미 생성된 AI 답변은 변경하지 않는다.
- 사용자 또는 대화 삭제 시 보관 기간을 기다리지 않고 관련 파일을 즉시 삭제한다.
- 업로드 파일은 실행 가능한 웹 경로 밖에 저장한다.
- 사용자별 저장 경로 또는 객체 키를 분리하고 다운로드 시 소유권을 검사한다.
- 파일명은 서버 내부 저장명으로 사용하지 않고 안전한 식별자를 생성한다.
- 악성 파일 검사 도입 여부는 **검토 필요**이다.
- 모델 API에 첨부파일 또는 추출 내용이 전달된다는 사실을 사용자에게 안내한다.

## 8. 모델 파라미터 설정

사용자는 API와 모델이 지원하고 관리자가 허용한 항목만 변경할 수 있다. 변경값은 대화방 단위로 저장되며 다른 대화방의 설정을 변경하지 않는다.

초기 지원 후보는 다음과 같다.

- `temperature`
- `top_p`
- 최대 출력 토큰 수
- `frequency_penalty`
- `presence_penalty`
- `seed`(지원 모델 한정)
- system prompt 또는 사용자 지침
- 이전 대화 턴 수(`0`은 무제한)
- 컨텍스트 토큰 목표값(기본 100,000)
- 첨부문서 후속 메시지 포함 여부

각 파라미터는 다음 정보를 갖는다.

- 기본값
- 최솟값과 최댓값
- 사용자 변경 허용 여부
- 해당 공급자 및 모델의 지원 여부

서버는 사용자가 보낸 값을 그대로 API에 전달하지 않고 허용 목록, 자료형 및 범위를 다시 검증한다. API 키, 엔드포인트, 모델 권한, 도구 호출 등 보안과 관련된 값은 일반 사용자가 변경할 수 없다.

## 12. Linux 서버 운영 요건

- Linux 서버에서 컨테이너 방식으로 실행할 수 있게 구성하는 것을 권장한다.
- 애플리케이션은 기본적으로 `127.0.0.1:32432`에서 수신하고 시작 설정 파일의 `server.port`로 변경할 수 있게 한다.
- 기존 Nginx가 외부 80·443 port와 HTTPS를 담당하고 애플리케이션으로 reverse proxy한다.
- port, 관리자 ID·password hash·TOTP와 구동 필수값은 서버 구동 파일 폴더의 `config.yaml`에서 시작 시 읽는다.
- HTTPS를 필수로 적용한다.
- 애플리케이션, 데이터베이스, 업로드 저장소를 분리한다.
- 운영 비밀값은 환경변수 또는 secret 관리 기능으로 주입한다.
- 데이터베이스와 첨부파일을 정기적으로 백업한다.
- 로그 순환, 디스크 사용량 감시 및 장애 알림을 적용한다.
- 외부 AI API 요청에 연결·응답 제한시간과 재시도 정책을 적용한다.
- 동시 요청 수와 사용자별 요청 빈도를 제한한다.

초기 소규모 서비스의 권장 구성 예시는 다음과 같다.

- 웹 애플리케이션: Next.js 또는 유사한 서버 포함 웹 프레임워크
- 데이터베이스: PostgreSQL
- 파일 저장: 서버 볼륨 또는 S3 호환 객체 저장소
- 리버스 프록시 및 HTTPS: 기존 Nginx
- 배포: Docker Compose

## 15. 인수 조건

- 공개 회원가입 경로가 없고 관리자만 계정을 생성할 수 있다.
- 비활성화된 계정은 로그인하거나 API 요청을 보낼 수 없다.
- 사용자는 허용되지 않은 모델을 화면과 직접 API 호출 모두에서 사용할 수 없다.
- 서로 다른 두 사용자가 상대방의 대화 ID나 첨부파일 ID를 알아도 내용을 조회할 수 없다.
- API 키는 브라우저 응답, 일반 로그 및 사용자 화면에 노출되지 않는다.
- TXT와 일반 PDF의 텍스트를 추출하여 질문에 활용할 수 있다.
- 지원되는 멀티모달 모델에서 이미지 내용을 바탕으로 답변할 수 있다.
- 파일은 10MB, 메시지당 10개, PDF 100페이지 제한이 서버에서 강제된다.
- 컨텍스트 한도를 넘으면 지정된 요약 모델과 프롬프트로 요약하고 원본 대화는 보존한다.
- 답변 재생성 결과가 분기로 저장되고 활성 분기만 다음 요청에 사용된다.
- 사용자는 같은 대화에서 모델을 변경할 수 있다.
- 4번째 로그인 시 가장 오래된 활성 세션이 종료된다.
- 같은 게스트 코드를 사용한 두 접속자가 서로의 대화와 첨부파일을 조회할 수 없다.
- 게스트와 일반 사용자의 일일 호출 제한은 동시 요청에서도 설정값을 넘지 않는다.
- 허용 범위를 벗어난 파라미터 요청은 서버가 거부한다.
- 응답을 스트리밍으로 표시하고 사용자가 생성을 중단할 수 있다.
- Linux 서버에서 문서화된 절차로 설치, 실행, 백업 및 복구할 수 있다.

일일 호출 횟수 제한의 정책·원자적 예약 기반은 게스트·모델 권한 단계에서 구현했으며 실제 AI 호출 endpoint가 추가될 때 upstream 전송 직전에 연결한다. token·비용 기반 예산 제한과 관리자 대화 본문 열람은 구조만 확장 가능하게 만들고 실제 UI와 정책은 2차 범위로 미룬다.

## 17. 검수 의견

제시된 사양은 구현 가능한 범위이며, 개인용 채팅을 넘어 소규모 다중 사용자 서비스로 분류된다. 개발 난이도를 좌우하는 핵심은 여러 API의 규격 차이, 사용자 데이터 격리, API 키 보호, 파일 처리 및 사용량 통제이다.

초기 버전은 제공자 템플릿 기반의 빠른 등록과 LLM Gateway를 통한 GPT·Gemini·Claude 사용을 우선 지원한다. 제공자별 세부 설정은 고급 설정에서 제공하며 token·비용 기반 제한·관리자 본문 열람 및 대용량 문서 검색은 기본 구조가 검증된 후 확장한다.
