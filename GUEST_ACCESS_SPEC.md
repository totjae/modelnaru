# ModelNaru 게스트 체험 명세

## 1. 목적

회원가입이나 공유 사용자 계정 없이 외부 시험자가 ModelNaru의 실제 대화 기능을 체험하게 한다. 하나의 게스트 코드를 여러 사람에게 공유하더라도 서버는 접속자마다 별도 임시 주체와 세션을 발급하여 대화, 메시지, 첨부파일과 컨텍스트를 서로 격리한다.

## 2. 적용 범위

- 게스트 체험 기능의 활성화, 공유 코드, 허용 모델과 호출 제한을 관리자가 설정한다.
- 비로그인 포트폴리오에서 서버·Docker·Nginx 구성과 역할별 기능·게스트 정책을 확인한 뒤 페이지 마지막의 공유 코드 입력으로 독립 임시 세션에 참가한다.
- 일반 사용자 계정, 회원가입과 익명 무제한 접속은 만들지 않는다.
- 실제 AI 대화, 첨부파일과 사용량 집계는 각 기능 구현 시 이 명세의 권한·수명·제한을 강제한다.
- 게스트는 관리자가 명시적으로 허용한 기능만 사용할 수 있다.

## 3. 역할과 격리

### 3.1 게스트 주체

게스트 코드 인증에 성공할 때마다 추측 불가능한 임시 `guest_id`와 서버 세션을 새로 만든다. `guest_id`는 사용자 계정이 아니며 사용자명, 비밀번호, 이메일과 복구 기능을 갖지 않는다.

- 서로 다른 브라우저나 브라우저 profile은 서로 다른 게스트 주체가 된다.
- 같은 브라우저의 여러 탭은 같은 유효 세션과 임시 작업공간을 공유한다.
- 유효한 게스트 세션이 있는 브라우저에서 다시 참가하면 기존 세션을 사용하고 불필요한 새 주체를 만들지 않는다.
- 게스트는 일반 사용자·다른 게스트·관리자의 ID, 대화, 메시지, 첨부파일과 사용량을 조회할 수 없다.
- 모든 게스트 소유 데이터 조회는 session의 `guest_id`를 DB 조건에 포함한다. 클라이언트가 보낸 `guest_id`는 신뢰하지 않는다.

### 3.2 허용 기능

게스트는 다음 기능 중 관리자가 활성화한 항목만 사용할 수 있다.

- 허용된 Provider 모델 선택과 텍스트 대화
- 허용 범위 내 모델 파라미터 변경
- 임시 대화방 생성·조회·삭제
- 파일 기능 구현 후 허용된 형식의 임시 첨부파일 사용
- 답변 생성 중단과 재생성

게스트는 사용자 계정 관리, Provider·API 키, 다른 주체의 데이터, 관리자 API와 로그에 접근할 수 없다. API 키, 연결 이름과 내부 Provider endpoint는 게스트 응답에 포함하지 않는다.

## 4. 참가와 세션

### 4.1 참가 흐름

1. 관리자가 게스트 체험을 활성화하고 공유 게스트 코드를 설정한다.
2. 로그인 화면은 활성화 여부를 조회해 `게스트로 체험하기`를 표시한다.
3. 시험자가 공유 코드를 입력한다.
4. 서버는 코드 hash, 생성 빈도, 전체 동시 세션 한도와 차단 상태를 검증한다.
5. 성공하면 독립 `guest_id`, 게스트 session과 CSRF token을 발급한다.
6. 게스트용 기본 모델·권한·남은 호출량을 반환하고 임시 작업공간으로 이동한다.

게스트 코드는 Argon2id로 hash하여 저장하고 등록 이후 원문을 다시 표시하지 않는다. 코드는 최소 6자, 최대 128자로 하고 관리자가 변경할 수 있다. 게스트 설정을 저장하면 코드 변경 여부와 관계없이 기존 게스트 세션을 모두 종료하며, 이후 참가에는 저장된 최신 설정과 코드가 적용된다.

### 4.2 세션 수명

- 기본 idle 만료: 마지막 정상 활동 후 1시간
- 기본 absolute 만료: 발급 후 24시간
- 관리 가능 범위: idle 15분~~6시간, absolute 1~~72시간
- session cookie: `HttpOnly`, `Secure`, `SameSite=Lax`, host-only, `Path=/`
- CSRF cookie와 상태 변경 header 검증은 일반 사용자와 동일하게 적용한다.
- 로그아웃하면 세션을 즉시 폐기하고 해당 게스트의 임시 데이터를 삭제 대상으로 표시한다.
- 만료된 세션과 임시 데이터는 정리 작업이 늦어도 1시간 이내 삭제한다.

게스트 세션은 일반 사용자의 계정당 최대 3세션 규칙과 별개다. 기본 전체 활성 게스트 세션은 10개이며 관리자가 1~100 범위에서 변경할 수 있다. 한도 도달 시 기존 세션을 임의 종료하지 않고 새 참가를 `GUEST_CAPACITY_REACHED`로 거부한다.

## 5. 관리자 설정

게스트 설정은 DB의 singleton 설정으로 관리하고 기본값은 비활성화다.

| 설정                         | 기본값       | 허용 범위·의미                               |
| ---------------------------- | ------------ | -------------------------------------------- |
| 체험 기능 활성화             | `false`      | 비활성화하면 새 참가 거부                    |
| 게스트 코드                  | 미설정       | Argon2id hash만 저장, 활성화 전에 필수       |
| 최대 활성 게스트 세션        | 10           | 1~100                                        |
| session당 일일 호출 제한     | 20           | 1~1,000                                      |
| 전체 게스트 일일 호출 제한   | 100          | 1~100,000                                    |
| idle 만료                    | 60분         | 15~360분                                     |
| absolute 만료                | 24시간       | 1~72시간                                     |
| 일일 집계 timezone           | `Asia/Seoul` | IANA timezone, 날짜 경계 계산에 사용         |
| 파일 첨부                    | `false`      | 파일 기능 구현 후 관리자가 명시적으로 활성화 |
| session 만료 데이터 삭제기한 | 1시간        | 만료·로그아웃 후 정리 완료 목표              |
| 전송 기록 허용               | 사용         | session 메모리에서 대화별 최대 3개           |

관리자는 다음 작업을 수행할 수 있다.

- 체험 기능 활성화·비활성화
- 게스트 코드 설정·교체
- 허용 모델과 모델별 파라미터 정책 설정
- session당·전체 일일 호출 제한과 session 수명 설정
- 현재 활성 session 수와 집계 사용량 조회
- 개별 또는 전체 게스트 session 강제 종료

체험 기능을 비활성화하면 새 참가는 즉시 차단한다. 기존 session을 유지할지 전부 종료할지는 관리자 변경 요청에서 명시적으로 선택하며 기본 동작은 유지다.

## 6. 모델 권한과 호출 횟수 제한

### 6.1 일반 사용자

- 사용자 계정 전체의 일일 호출 제한을 설정할 수 있다. `null`은 무제한이다.
- `user_model_permissions`에서 모델 허용 여부, 모델별 일일 호출 제한과 파라미터 정책을 설정한다.
- 계정 전체와 선택 모델 제한 중 하나라도 소진되면 요청을 거부한다.

### 6.2 게스트

- 게스트는 `guest_model_permissions`에 허용된 활성 모델만 볼 수 있다.
- session당 제한, 전체 게스트 제한과 선택 모델 제한을 모두 검사한다.
- 새 session을 반복 생성해 session당 제한을 우회할 수 있으므로 전체 게스트 제한과 IP별 생성 제한을 항상 함께 적용한다.

### 6.3 집계 규칙

- 날짜 경계는 설정한 IANA timezone의 현지 날짜를 사용한다.
- 사용자가 메시지를 보내거나 답변을 재생성해 하나의 논리적 AI 생성을 시작할 때 1회로 계산한다.
- 입력 검증·권한 검증 단계에서 거부된 요청은 계산하지 않는다.
- upstream으로 전송하기 직전에 DB transaction으로 한도를 확인하고 사용량을 원자적으로 예약한다.
- upstream 오류나 사용자 취소가 발생해도 이미 전송한 요청은 차감한다.
- 동일 논리 요청의 자동 재시도는 추가 차감하지 않으며 별도 요청 log에는 실제 upstream 시도 횟수를 기록한다.
- 관리자가 한도를 낮춰 이미 사용한 횟수보다 작아지면 추가 요청만 차단하며 기존 집계를 수정하지 않는다.
- 권한 회수와 한도 변경은 다음 요청부터 즉시 적용한다.

현재 한도 예약 오류는 HTTP `429`와 `ACCESS_DAILY_LIMIT_REACHED`를 사용하고 범위를 `user`, `user_model`, `guest_session`, `guest_model`, `guest_global` 중 하나로 반환한다. 날짜 경계의 `resetAt`과 남은 횟수 조회는 실제 AI 요청 API를 연결할 때 추가하며 다른 사용자의 사용량은 포함하지 않는다.

## 7. 데이터 구조

`0004_access_and_guest.sql`에서 다음 구조를 확정했다.

- `guest_settings`: 기능 상태, 코드 hash, session·호출 제한, timezone과 만료 정책
- `guest_principals`: 임시 주체, 생성·마지막 활동·만료·삭제 상태
- `sessions.guest_id`: `principal_type = 'guest'`일 때만 설정되는 FK
- `guest_model_permissions`: 허용 모델, 모델별 일일 제한과 파라미터 정책
- `daily_usage_counters`: 현지 날짜, 주체 범위, 모델과 예약 호출 수를 원자적으로 집계
- `conversations.guest_id`: 일반 사용자 소유권과 상호 배타적인 임시 소유권
- 첨부파일과 메시지: 대화 소유권을 통해 동일 게스트 범위를 상속

일반 사용자 삭제와 게스트 정리 시 FK cascade 또는 같은 transaction으로 대화, 메시지, 요약과 첨부파일 metadata를 삭제한다. 원본 파일은 DB 삭제와 함께 정리 queue에 넣고 재시도 가능한 삭제 상태를 기록한다.

## 8. API

정확한 request·response는 [API_SPEC.md](./API_SPEC.md)에서 관리한다. 현재 endpoint는 다음과 같다.

- `GET /api/auth/guest/status`: 비로그인 화면에 활성 여부만 반환
- `POST /api/auth/guest/session`: 공유 코드 검증과 독립 session 발급
- `POST /api/auth/logout`: 일반 사용자와 동일한 logout endpoint 사용
- `GET /api/access/models`: 일반 사용자·게스트에게 허용된 활성 모델 조회
- `GET /api/admin/access`: 사용자·게스트 접근 정책 조회
- `PUT /api/admin/access/users/:id`: 사용자 모델 권한·일일 제한 변경
- `PUT /api/admin/access/guest`: 게스트 정책·코드·모델 권한을 변경하고 기존 게스트 session 전체 종료

개별 게스트 session 목록·종료와 남은 호출량 조회는 대화·관리자 로그 단계에서 추가한다.

비로그인 status API는 코드 hash, 제한값과 내부 설정을 노출하지 않는다. 관리자 상태 변경 API는 관리자 session, CSRF와 감사 로그를 요구한다.

## 9. 보안·오류·경계 조건

- 게스트 기능은 코드가 설정되지 않으면 활성화할 수 없다.
- 코드 비교 결과와 존재 여부는 외부 오류 메시지에서 구분하지 않는다.
- 코드 실패는 IP hash 기준 기본 5회/15분, session 생성은 기본 5회/시간으로 제한한다.
- 원본 IP는 애플리케이션 DB log에 저장하지 않고 서버 secret으로 HMAC한 값을 사용한다.
- `X-Forwarded-For`는 신뢰 proxy 설정과 실제 Nginx 경로가 일치할 때만 사용한다.
- 새 참가에는 전체 활성 session 한도와 DB의 원자적 생성 검사를 적용한다.
- 게스트 코드는 봇 방지를 완전히 보장하지 않는다. 공개 범위 확대 시 CAPTCHA는 후속 선택 기능으로 둔다.
- 대화 본문, system prompt, 첨부 내용과 API 키는 인증·보안·AI 요청 log에 기록하지 않는다.
- 게스트에게 사용자 ID 존재 여부, Provider 자격증명과 관리자 설정을 노출하지 않는다.
- 정리 작업 실패 시 session 접근은 계속 차단하고 삭제를 재시도하며 시스템 경고를 남긴다.

표준 오류 code는 다음과 같다.

- `GUEST_DISABLED`: 체험 기능 비활성
- `GUEST_AUTH_FAILED`: 잘못된 코드 또는 인증 실패
- `GUEST_RATE_LIMITED`: 코드 시도·session 생성 빈도 초과
- `GUEST_CAPACITY_REACHED`: 전체 활성 session 한도 도달
- `GUEST_SESSION_EXPIRED`: idle 또는 absolute 만료
- `GUEST_MODEL_FORBIDDEN`: 허용되지 않거나 비활성인 모델
- `ACCESS_DAILY_LIMIT_REACHED`: 적용되는 일일 호출 제한 소진

## 10. 로그와 개인정보

다음 이벤트를 대화 본문 없이 기록한다.

- 게스트 참가 성공·실패·속도 제한
- session 만료·logout·관리자 강제 종료
- 게스트 설정·코드·모델 권한·호출 제한 변경
- 일일 제한 거부와 적용 범위
- 임시 데이터 정리 건수·실패

게스트 코드는 원문·hash 모두 감사 snapshot과 일반 log에서 제외한다. 관리 화면은 무작위 guest ID의 축약값, session 생성·만료 시각, 호출 집계와 상태만 표시한다. IP hash는 관리자 화면과 내보내기에 기본 표시하지 않는다.

관리자가 `게스트 전송 기록 허용`을 켠 경우에만 게스트가 대화별 0~3개의 실제 Provider 요청·응답을 현재 session에서 확인할 수 있다. 기록은 session 메모리에만 존재하고 코드·API key·이미지 base64는 제거된다. 게스트 설정을 저장하면 기존 게스트 session과 전송 기록을 모두 종료·삭제한다.

## 11. 검증·인수 조건

- 게스트 체험이 꺼져 있어도 포트폴리오의 시스템 구성·역할별 기능·게스트 정책 설명은 표시되고 코드 입력과 참가 버튼만 비활성화된다.
- 같은 코드를 사용한 두 브라우저가 서로 다른 `guest_id`를 받고 상대 대화 ID를 알아도 조회할 수 없다.
- 새로고침과 같은 브라우저 탭에서는 유효한 session과 임시 대화가 유지된다.
- idle·absolute 만료, logout과 관리자 종료 후 기존 session 요청이 거부된다.
- 만료·logout 후 게스트 대화와 첨부 데이터가 정해진 기한 안에 삭제된다.
- session당·모델별·전체 게스트 일일 제한이 동시 요청에서도 초과되지 않는다.
- 일반 사용자 계정 전체·모델별 일일 제한도 같은 원자적 규칙을 사용한다.
- 비활성·권한 회수된 모델은 다음 요청부터 즉시 거부된다.
- 게스트 코드, API 키, 대화 본문과 원본 IP가 response·일반 log·감사 snapshot에 노출되지 않는다.
- 게스트 session으로 관리자와 일반 사용자 전용 API에 접근할 수 없다.
- 게스트 전송 기록은 현재 session에서만 보이고 설정 저장·logout·만료 뒤 즉시 사라진다.
