# ModelNaru Web UI 명세

## 1. 목적

ModelNaru의 로그인, 사용자 작업공간, 채팅과 관리자 화면이 같은 시각 언어를 사용하도록 공통 UI 기준을 정의한다.

## 2. 적용 범위

Next.js Web의 모든 화면, 공통 컴포넌트, 반응형 레이아웃과 상태 표현에 적용한다.

## 3. 상세 명세

### 3.1 기본 콘셉트

- 다크 테마는 검정, 라이트 테마는 흰색과 무채색을 기반으로 한다.
- 넓은 면적에는 녹색이나 7색 혼합 gradient 배경을 사용하지 않는다.
- 정보 밀도가 높은 도구형 UI를 지향하되 충분한 여백과 명확한 계층을 유지한다.
- 7색은 각 기능과 상태에 하나씩 배정하는 브랜드 포인트다. 한 요소에는 원칙적으로 한 가지 포인트 색만 사용한다.
- 7색을 섞은 무지개 gradient는 사용하지 않는다.
- 본문과 넓은 카드 배경에는 유채색을 채우지 않는다.

### 3.2 색상 토큰

| 용도        | 라이트                                   | 다크                        |
| ----------- | ---------------------------------------- | --------------------------- |
| 전체 배경   | `#ffffff`                                | `#050505`                   |
| 기본 표면   | `rgba(255, 255, 255, 0.96)`              | `rgba(14, 14, 15, 0.94)`    |
| 강한 표면   | `#f7f7f8`                                | `#101011`                   |
| 기본 테두리 | `rgba(0, 0, 0, 0.14)`                    | `rgba(255, 255, 255, 0.13)` |
| 기본 글자   | `#111114`                                | `#f5f5f7`                   |
| 보조 글자   | `#686870`                                | `#9a9aa2`                   |
| 기본 강조   | `#111114`                                | `#f4f4f5`                   |
| 오류        | `#c93636`                                | `#ff8f8f`                   |
| 7색 순서    | 빨강, 주황, 노랑, 초록, 청록, 파랑, 보라 | 라이트 테마와 동일          |

7색은 각각 독립된 공통 token으로 관리한다.

| 색상 | 기본 역할                        |
| ---- | -------------------------------- |
| 빨강 | 삭제, 실패, 위험                 |
| 주황 | 관리자, 주의                     |
| 노랑 | 게스트                           |
| 초록 | 성공, 정상 동작                  |
| 청록 | 일반 사용자·사용자 관리          |
| 파랑 | 기본 동작, 접근 설정, 공통 focus |
| 보라 | 브랜드, Provider, 테마 설정      |

새 화면도 의미가 같은 기존 색을 재사용한다. 장식 목적으로 여러 색을 한 요소에 겹치거나 화면마다 역할을 바꾸지 않는다.

### 3.3 테마 동작

- 선택지는 `시스템`, `라이트`, `다크` 세 가지다.
- 최초 접속은 운영체제의 `prefers-color-scheme`을 따른다.
- 사용자가 고른 모드는 브라우저 local storage에 저장하고 다음 방문에도 적용한다.
- `시스템` 선택 중에는 운영체제 테마가 바뀌면 화면도 즉시 전환한다.
- 첫 화면 표시 전에 저장된 테마를 적용해 밝은 화면이 순간적으로 번쩍이는 현상을 방지한다.

### 3.4 컴포넌트

- 기본 버튼은 밝은 무채색 배경과 검정 글자를 사용한다.
- 보조 버튼은 중성 표면과 테두리를 사용하며 hover 때 해당 기능에 배정된 단색 테두리를 표시할 수 있다.
- 입력창은 중성 표면과 테두리를 사용하고 focus 상태에서 해당 기능의 단색 테두리를 표시한다.
- 카드의 기본 테두리는 중성색이며 로그인 카드, 현재 선택된 항목 등 의미 있는 곳에만 역할별 단색 테두리를 사용한다.
- 성공 상태는 흰색과 중성 표면을 기본으로 하고, 오류·위험 상태에는 제한적으로 빨강을 사용한다.
- 체크박스와 radio는 고정 크기로 정렬하며 주변 입력창의 전체 너비 규칙을 상속하지 않는다.

### 3.5 레이아웃

- 데스크톱은 관리·설정 영역을 2열까지 사용하고 860px 이하에서는 1열로 전환한다.
- 채팅 화면은 대화 목록, 대화 본문, 모델·파라미터 설정의 계층을 유지하되 작은 화면에서는 보조 영역을 drawer로 전환한다.
- 장시간 대화 사용을 고려해 본문 영역에는 강한 gradient나 지속적인 광원 효과를 사용하지 않는다.
- 데스크톱 채팅은 대화 목록·메시지 본문·대화 설정의 3열이며, 860px 이하에서는 설정을 하단으로 보내고 520px 이하에서는 전체를 1열로 전환한다.
- 사용자 메시지는 청록, AI 메시지와 생성 parameter는 보라, 전송은 청록, 중지는 빨강의 단색 역할을 사용한다.
- 기존 대화를 선택하거나 새로고침하면 활성 분기의 마지막 AI 응답에 저장된 모델을 선택창에 복원한다. 현재 권한에서 제외된 모델은 복원하지 않는다.
- 완료·실패·취소 AI 답변에는 재생성 버튼을 제공한다. 재생성 중에는 새 답변을 기존 위치에서 미리 보여주고 성공 후 별도 분기로 전환한다.
- 선택 가능한 분기가 둘 이상이면 원본과 재생성 답변을 전환하는 분기 선택기를 표시한다. 실패·취소된 재생성 분기는 선택기에 표시하지 않는다.

## 4. 오류·예외와 경계 조건

- 포인트 색상만으로 오류·성공·비활성 상태의 의미를 전달해서는 안 된다.
- 텍스트 대비는 WCAG AA를 목표로 하며 색상만으로 상태를 전달하지 않는다.
- 포인트 색상에 지속적인 점멸이나 색상 순환 애니메이션을 사용하지 않는다.
- 비활성 요소는 흐리게 표시하되 글자를 읽을 수 있는 대비를 유지한다.

## 5. 검증·인수 조건

- 다크에서는 검정, 라이트에서는 흰색 기반 표면이 사용되고 두 모드 모두 역할별 단색 포인트를 유지한다.
- 시스템 테마 추종, 수동 전환과 선택값 복원이 동작한다.
- 포인트 색상이 기능별 7색 역할표와 일치하고 무지개 gradient가 없다.
- 키보드 focus가 명확하며 체크박스 정렬이 입력창 너비 규칙에 영향받지 않는다.
- 860px와 520px 반응형 구간에서 가로 스크롤 없이 주요 기능을 사용할 수 있다.
- 모델을 바꿔 메시지를 보낸 뒤 새로고침하거나 다른 대화를 왕복해도 대화별 마지막 사용 모델이 복원된다.
- 답변을 재생성해도 원본을 다시 선택할 수 있고 분기 전환 후 새로고침해도 활성 분기가 유지된다.
- production build와 정적 검사가 통과한다.

## 6. 미결정·보류 항목

- 최종 로고와 아이콘 세트
- 채팅 화면의 정확한 3열 너비와 모바일 drawer 동작
- 채팅의 tool call·첨부파일 등에 배정할 세부 보조색
